Claude Code Integration

Persistent memory for Claude Code agents

Build agents in Agent OS. Run them in Claude Code. Memory follows everywhere. Same Letta-backed blocks, archival search, and team context — accessible via MCP.

What this solves

Claude Code's native agents use flat MEMORY.md files. No structure, no search, no shared state across teams. Agent OS fixes this.

Structured memory

Persona block, scratchpad, memory instructions — each with character limits and purpose. Not one flat file.

Shared team blocks

Decisions, task board, brand guide — visible to all team members. Update from one agent, every agent sees it.

Semantic search

Archival memory with vector search. Store craft knowledge, retrieve by meaning. Unlimited storage.

Auto-categorization

Project decisions, user preferences, craft knowledge — sorted into the right block automatically on session end.

Setup

Five minutes from zero to persistent-memory agents in Claude Code.

1

Start Agent OS

Agent OS is the web UI where you build and manage agents.

cd agent-builder
npm install
npm run dev                # http://localhost:3000
docker compose up -d       # Letta server (optional but recommended)
2

Build your agents

Use the 6-stage builder or quick-create. Define mission, identity, capabilities, memory, triggers, guardrails.

3

Deploy to Letta

Click "Deploy" in the workspace. This creates the Letta agent with memory blocks. If you're using teams, deploy the team to create shared blocks.

4

Export to Claude Code

For teams, use the API. For individual agents, use the agent endpoint.

# Export a team (generates all files)
curl -X POST http://localhost:3000/api/teams/{teamId}/generate-claude-code

# Export a single agent definition
curl http://localhost:3000/api/agents/{agentId}/claude-code-agent

Copy the generated files to your project directory.

5

Add the MCP server

This gives Claude Code access to Agent OS memory.

# Option A: CLI
claude mcp add agent-os -s project -- npx agent-os-mcp --url http://localhost:3000

# Option B: .mcp.json (checked into project)
{
  "mcpServers": {
    "agent-os": {
      "command": "npx",
      "args": ["agent-os-mcp", "--url", "http://localhost:3000"]
    }
  }
}

That's it. Open Claude Code in your project directory. Your agents appear in /agents. They load memory from Agent OS on every session start.

The loop

  1. Build agents in Agent OS (conversational builder, memory config, team setup)
  2. Deploy to Letta (creates agents with structured memory blocks)
  3. Export to Claude Code (generates subagent definitions, MCP config, hooks)
  4. Work in Claude Code (agents load context, use memory tools, code normally)
  5. Sync automatically (SubagentStop hooks push session data back to Agent OS)
  6. Review in Agent OS (see updated memory, tweak configs, re-export)

How memory works

The problem

When agents run through Letta directly (Agent OS web UI), memory is automatic. Letta provides native tools and the model sees its memory on every turn.

When agents run in Claude Code, Letta is NOT in the inference loop. The MCP server bridges this gap.

CapabilityLetta native (web UI)Claude Code (via MCP)
Memory replace core_memory_replace (built-in) mcp__agent-os__core_memory_replace
Archival insert archival_memory_insert (built-in) mcp__agent-os__archival_insert
Memory in context Injected automatically by Letta load_context tool at session start
Categorization Prompt in system message Same prompt in subagent .md
Fallback sync N/A (always in loop) sync_session + server-side extraction

Memory scopes

Global (per agent)

  • Persona (5000 chars) — identity, preferences, rules
  • Scratchpad (5000 chars) — working notes
  • Archival (unlimited) — craft knowledge, skills

Project (shared across team)

  • Decisions (8000 chars) — requirements, constraints
  • Task board (6000 chars) — sprint tasks, blockers
  • Project (2000, read-only) — overview, goals
  • Brand (10000, read-only) — design system

Server-side extraction

Since Claude Code agents are focused on coding, they may not call memory tools reliably. The server-side fallback handles this:

  1. sync_session sends a summary to Agent OS
  2. Agent OS calls Claude with an extraction prompt
  3. Learnings are categorized: decisions, preferences, knowledge, task updates
  4. Each category is written to the appropriate Letta block
  5. Next session, load_context returns the updated memory

MCP tools reference

Eight tools available via the agent-os MCP server.

ToolPurposeWhen to call
load_context Full agent identity + memory blocks + team context Session start
get_memory_blocks Read current memory block values Anytime
core_memory_replace Find-and-replace text in a block Correct a fact, update a preference
core_memory_append Append text to a block Log a decision, add a task update
archival_search Semantic search over archival memory Recall past knowledge
archival_insert Store in long-term archival memory Save craft knowledge, patterns
get_team_context Team members, roles, shared blocks Understand team state
sync_session Send summary for server-side extraction Session end (or via hook)

Tip: You don't need to call every tool. The memory protocol in each subagent's .md file tells the agent when to use each tool. The SubagentStop hook handles sync_session automatically.

Troubleshooting

"Agent has no Letta deployment"

The agent hasn't been deployed to Letta yet. In Agent OS, go to the agent workspace and click "Deploy". Make sure LETTA_BASE_URL is set in .env and docker compose up -d is running.

"Letta not configured"

The Agent OS server doesn't have Letta enabled. Set LETTA_BASE_URL=http://localhost:8283 in your .env file and restart the dev server.

MCP server not connecting

Check that Agent OS is running (npm run dev). Verify the URL: npx agent-os-mcp --url http://localhost:3000. Check Claude Code's MCP server logs.

Memory not syncing after session

Verify the SubagentStop hook is in .claude/settings.json. Check that the agent has a lettaAgentId (visible in the API response). Test manually: curl -X POST http://localhost:3000/api/agents/{id}/sync-session -d '{"summary":"test"}'

Agent doesn't load context on start

The memory protocol in the subagent .md instructs the agent to call load_context. If the agent isn't doing this, check that mcp__agent-os__* is in the tools list in the YAML frontmatter.

Works with any MCP client

The MCP server is runtime-agnostic. Agent OS owns identity and memory. The runtime owns execution.
